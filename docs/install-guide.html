Tuva FHIR-to-OMOP Transformer

Install this Snowflake Native App to transform FHIR R4 bundles into OMOP CDM v5.4 tables — entirely within your Snowflake account.

Before you begin

Prerequisites

ACCOUNTADMIN role Required to install apps and grant privileges
A warehouse Any size — XS works fine for testing
FHIR R4 data Table with a VARIANT column containing FHIR bundles
Account shared Ask Todd to add your account locator to the package
Performance: 1,000 FHIR bundles processes in ~65 seconds on an X-Small warehouse. Any size works — larger warehouses scale linearly.

1Install the application

This creates the app in your account from the shared package.

-- Install the app
CREATE APPLICATION TUVA_FHIR_TO_OMOP_APP
    FROM APPLICATION PACKAGE TUVA_FHIR_TO_OMOP_PKG;

2Grant privileges

The app needs permission to create its output database, use your warehouse, and read your source data.

-- Allow the app to create the OMOP output database
GRANT CREATE DATABASE ON ACCOUNT TO APPLICATION TUVA_FHIR_TO_OMOP_APP;

-- Let the app use your warehouse (replace with yours)
GRANT USAGE ON WAREHOUSE MY_WAREHOUSE TO APPLICATION TUVA_FHIR_TO_OMOP_APP;

-- Grant the app read access to your FHIR source data
GRANT USAGE ON DATABASE MY_DB TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
GRANT USAGE ON SCHEMA MY_DB.MY_SCHEMA TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
GRANT SELECT ON TABLE MY_DB.MY_SCHEMA.MY_FHIR_BUNDLES TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
Important: Replace MY_DB, MY_SCHEMA, MY_FHIR_BUNDLES, and MY_WAREHOUSE with your actual names in all commands on this page.

3Bind your FHIR source table

Register the table reference so the app knows where to find your FHIR bundles.

CALL TUVA_FHIR_TO_OMOP_APP.CORE.REGISTER_REFERENCE(
    'fhir_source_database', 'SET',
    'MY_DB.MY_SCHEMA.MY_FHIR_BUNDLES'
);

4Bind your warehouse

Tell the app which warehouse to use for compute.

-- Replace with your warehouse name
CALL TUVA_FHIR_TO_OMOP_APP.CORE.REGISTER_REFERENCE(
    'consumer_warehouse', 'SET',
    'MY_WAREHOUSE'
);

5Open the app

Navigate to Apps → TUVA_FHIR_TO_OMOP_APP in Snowsight. The Streamlit UI will guide you through:

  • Configure — Select your source table, JSON column, and output schema
  • Run — Execute the full FHIR → OMOP pipeline or individual mappers
  • Vocabulary — Browse the included terminology mappings (LOINC, SNOMED, RxNorm, etc.)
  • Coverage — See how many records mapped to standard OMOP concepts
  • Explore — Browse the output OMOP CDM tables
Tip: The default output schema is OMOP_STAGING. You can change it in the Configure tab. If you select OMOP_CDM, the app will warn you and show existing row counts before allowing overwrites.

Expected FHIR table format

Your source table should have at minimum:

CREATE TABLE my_fhir_bundles (
    BUNDLE_ID   TEXT,          -- unique ID per bundle
    BUNDLE_DATA VARIANT       -- the FHIR R4 Bundle JSON
);

Column names are configurable in the app — BUNDLE_ID and BUNDLE_DATA are defaults.

Don't have FHIR data? Generate synthetic bundles with Synthea, then load the JSON files into a VARIANT column.

OMOP tables created

The pipeline produces these OMOP CDM v5.4 tables in your chosen output schema:

  • person — Demographics (gender, birth date, race, ethnicity)
  • visit_occurrence — Encounters / visits
  • condition_occurrence — Diagnoses
  • measurement — Labs, vitals, observations
  • drug_exposure — Medications
  • procedure_occurrence — Procedures
  • observation — Qualitative observations (smoking status, etc.)
  • death — Mortality records

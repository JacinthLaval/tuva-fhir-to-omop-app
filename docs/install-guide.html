Snowflake Health Data Forge

Transform HL7v2 messages, FHIR R4 JSON bundles, or FHIR R4 XML bundles into OMOP CDM v5.4 or Tuva Input Layer tables — entirely within your Snowflake account.

Before you begin

Prerequisites

ACCOUNTADMIN role Required to install apps and grant privileges
A warehouse Any size — XS works fine for testing
Clinical data HL7v2, FHIR R4 JSON, or FHIR R4 XML in a Snowflake table
Account shared Ask Todd to add your account locator to the package
Performance: 1,000 FHIR bundles in ~65s, 4,013 HL7v2 messages in ~270s on an X-Small warehouse. Larger warehouses scale linearly.
Supported formats

Input Formats (auto-detected)

HL7v2 Pipe-delimited messages (10 segment types)
FHIR R4 JSON VARIANT column with Bundle JSON
FHIR R4 XML VARCHAR column with Bundle XML
Choose your output

Output Formats

OMOP CDM v5.4 23 mappers producing 17 clinical + infrastructure tables. Concept IDs mapped via vocabulary crosswalks (LOINC, SNOMED, RxNorm, ICD-10, CPT, HCPCS).
Tuva Input Layer 13 mappers producing analytics-ready tables using source codes directly. Each row stamped with a data_source label.

1Install the application

This creates the app in your account from the shared package.

-- Install the app
CREATE APPLICATION TUVA_FHIR_TO_OMOP_APP
    FROM APPLICATION PACKAGE TUVA_FHIR_TO_OMOP_PKG
    USING VERSION V1_6;

2Grant privileges

The app needs permission to create its output database, use your warehouse, and read your source data.

-- Allow the app to create the output database
GRANT CREATE DATABASE ON ACCOUNT TO APPLICATION TUVA_FHIR_TO_OMOP_APP;

-- Let the app use your warehouse (replace with yours)
GRANT USAGE ON WAREHOUSE MY_WAREHOUSE TO APPLICATION TUVA_FHIR_TO_OMOP_APP;

-- Grant the app read access to your source data
GRANT USAGE ON DATABASE MY_DB TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
GRANT USAGE ON SCHEMA MY_DB.MY_SCHEMA TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
GRANT SELECT ON ALL TABLES IN SCHEMA MY_DB.MY_SCHEMA TO APPLICATION TUVA_FHIR_TO_OMOP_APP;
Important: Replace MY_DB, MY_SCHEMA, and MY_WAREHOUSE with your actual names in all commands on this page.

3Bind your source table

Register the table reference so the app knows where to find your clinical data.

CALL TUVA_FHIR_TO_OMOP_APP.CORE.REGISTER_REFERENCE(
    'fhir_source_database', 'SET',
    'MY_DB.MY_SCHEMA.MY_SOURCE_TABLE'
);

4Bind your warehouse

Tell the app which warehouse to use for compute.

CALL TUVA_FHIR_TO_OMOP_APP.CORE.REGISTER_REFERENCE(
    'consumer_warehouse', 'SET',
    'MY_WAREHOUSE'
);

5Open the app

Navigate to Apps → TUVA_FHIR_TO_OMOP_APP in Snowsight. The Streamlit UI will guide you through:

  • Configure — Select your source table, column, and output format (OMOP or Tuva)
  • Run — Execute the full pipeline or individual mappers with per-step progress
  • Vocabulary — Browse the included terminology mappings (LOINC, SNOMED, RxNorm, etc.)
  • Coverage — See how many records mapped to standard concepts
  • Explore — Browse the output tables (format-aware: OMOP or Tuva)
  • Quality — Run FHIR quality validation on your input data
  • History — View transformation run history and audit trail
Tip: The app auto-detects your input format (HL7v2, JSON, XML). The default OMOP output schema is omop_staging and Tuva output schema is tuva_input.

Expected source table formats

FHIR R4 JSON:

CREATE TABLE my_fhir_bundles (
    BUNDLE_ID   TEXT,          -- unique ID per bundle
    BUNDLE_DATA VARIANT       -- FHIR R4 Bundle JSON
);

FHIR R4 XML:

CREATE TABLE my_fhir_xml (
    BUNDLE_ID   TEXT,          -- unique ID per bundle
    BUNDLE_DATA TEXT          -- FHIR R4 Bundle XML string
);

HL7v2:

CREATE TABLE my_hl7v2_messages (
    MESSAGE_ID   TEXT,         -- unique ID per message
    MESSAGE_DATA TEXT         -- HL7v2 pipe-delimited message
);

Column names are configurable in the app.

Don't have clinical data? Generate synthetic FHIR bundles with Synthea, then load the JSON files into a VARIANT column.

OMOP CDM v5.4 output (23 mappers)

When choosing OMOP output, the pipeline produces these tables:

  • person — Demographics (gender, birth date, race, ethnicity)
  • visit_occurrence — Encounters / visits
  • condition_occurrence — Diagnoses
  • measurement — Labs, vitals, quantitative observations
  • drug_exposure — Medications, immunizations, administrations
  • procedure_occurrence — Procedures, imaging studies
  • observation — Qualitative observations, allergies, care plans, care teams
  • death — Mortality records
  • location — Physical locations
  • care_site — Organizations / facilities
  • provider — Practitioners
  • payer_plan_period / cost — Claims and costs
  • observation_period — Derived event windows per person
  • cdm_source — ETL metadata

Tuva Input Layer output (13 mappers)

When choosing Tuva output, the pipeline produces these analytics-ready tables:

  • patient — Demographics
  • encounter — Visits
  • condition — Diagnoses
  • lab_result — Lab values
  • observation — Clinical observations
  • medication — Prescriptions
  • immunization — Vaccines
  • procedure — Procedures
  • location — Locations
  • practitioner — Providers
  • medical_claim — Claims
  • eligibility — Coverage
  • appointment — Scheduled encounters
